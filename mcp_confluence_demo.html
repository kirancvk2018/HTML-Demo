Four inspection points, one build

Nothing lands in Confluence without passing four separate checkpoints — each one owned by a different party, each with its own sign-off.

Checkpoint 1 · Host ⇄ Client
One work crew per site
The host (Claude Desktop, Code, or an API app) keeps a dedicated MCP client — and connection — for Confluence, separate from Jira, Gmail, or any other connected site.
Checkpoint 2 · Client ⇄ Server
Local rig or remote plant
A local Confluence MCP server runs over stdio as a spawned subprocess. A hosted one runs Streamable HTTP over a session ID — same JSON-RPC drawings either way.
Checkpoint 3 · Server ⇄ Atlassian
The server carries the permit
Claude never holds your Atlassian credential. The MCP server holds a scoped OAuth 2.0 token (e.g. write:confluence-content) and presents it at the gate.
Checkpoint 4 · Result ⇄ Model
The as-built comes back the same route
Confluence's JSON response is reshaped into MCP tool-result content and travels back over the exact connection that carried the work order out.
