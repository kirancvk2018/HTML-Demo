Point 1 · Host ⇄ Client
One client per server
The host (Claude Desktop, Claude Code, or an API app) spins up a dedicated MCP client for each connected server — Jira, GitHub, Slack each get their own isolated session.
Point 2 · Client ⇄ Server
Transport choice matters
Local servers use stdio (spawned subprocess, no network). Remote servers use Streamable HTTP with a session ID — this is how the hosted Atlassian MCP server works.
Point 3 · Server ⇄ Jira
The MCP server holds the real credential
The model never sees a Jira token. The MCP server exchanges OAuth 2.1 (remote) or a stored API token (local) for access, then calls the REST API on the model's behalf.
Point 4 · Result ⇄ Model
Same pipe, reverse direction
The Jira JSON response is reshaped into MCP "tool result content" and travels back over the exact same transport connection that carried the request.
